Building Registration and Login Pages with Flask Forms

The application already supports authentication through JSON API endpoints at /auth/register, /auth/login, and /auth/logout. In this lesson we extend those same routes to also serve browser HTML forms, so both a REST API client and a regular browser user can share the same URL — without breaking any existing behavior.

01

Starting Point: What We Already Have

The current Flask application already supports basic authentication through API endpoints. A client can send credentials to the backend, the server can create a user, verify a password, and create an authenticated session. That is a strong backend foundation, but browser users usually expect pages: a registration page, a login page, visible error messages, and a navigation bar that changes when someone is logged in.

In this lesson, we are not adding email confirmation or a password reset flow. We are intentionally starting with simple authentication: create an account, log in, log out, and restrict selected routes to authenticated users. Once this foundation is working, more advanced security features can be layered on later.

The browser form flow: request page → submit form → validate → perform authentication action.

Conceptual Anchor
Authentication still means “Who are you?” The only thing changing today is the interface. Instead of only sending JSON from Postman or a frontend client, we are letting a browser user submit regular HTML forms.
02

Serving Both API and Form Clients

Rather than writing separate routes for API access and browser access, we can teach each route to detect which kind of client is making the request. When a client sends Content-Type: application/json, Flask sets request.is_json to True. A plain browser form submission uses application/x-www-form-urlencoded, so request.is_json will be False.

Inside POST, the route branches on that flag: the JSON path uses the existing request.get_json() logic and returns jsonify(...); the form path uses form.validate_on_submit() and returns a redirect or a re-rendered template. The database operations and the model are the same in both paths.

Concern JSON API path (request.is_json) Form path (browser)
HTTP methods POST only GET (render form) + POST (submit)
Read input request.get_json() FlaskForm reads submitted fields automatically
Validation Manual checks in the route form.validate_on_submit() with WTForms validators
Success response jsonify(user.to_dict()) + status code flash() + redirect()
Error response jsonify({"error": "..."}) + 4xx code flash() + render_template() with form errors
CSRF protection Not needed (no cookies involved) {{ form.hidden_tag() }} renders hidden token
Key Principle
The routes are extended, not replaced. Any existing API client — Postman, a React app, a mobile client — continues to work without any changes. The form support is purely additive.
How does request.is_json work under the hood?

Flask checks the Content-Type header of the incoming request. If it is application/json (or another JSON media type), request.is_json returns True. A browser submitting an HTML form sends application/x-www-form-urlencoded by default, so the value is False. The check is cheap and does not read the body.

Install and Configure Flask-WTF

Flask-WTF integrates WTForms with Flask. It gives us Python form classes, validators, automatic error collection, and CSRF protection. CSRF protection matters because browser forms automatically include cookies, so we want a hidden token that proves the form was generated by our application.

04

Create Form Classes

A form class describes expected fields and validation rules. This keeps validation close to the shape of the form, instead of scattering checks throughout the route. For registration, we need username, email, password, and password confirmation. For login, we need username, password, and optionally a remember-me checkbox.

app/auth/forms.py
python
from flask_wtf import FlaskForm
from wtforms import StringField, PasswordField, BooleanField, SubmitField
from wtforms.validators import DataRequired, Email, EqualTo, Length, ValidationError

from app.models import User


class RegistrationForm(FlaskForm):
    username = StringField(
        "Username",
        validators=[DataRequired(), Length(min=3, max=80)]
    )
    email = StringField(
        "Email",
        validators=[DataRequired(), Length(max=120), Email()]
    )
    password = PasswordField(
        "Password",
        validators=[DataRequired(), Length(min=8)]
    )
    confirm_password = PasswordField(
        "Confirm Password",
        validators=[DataRequired(), EqualTo("password", message="Passwords must match.")]
    )
    submit = SubmitField("Create Account")

    def validate_username(self, username):
        existing = User.query.filter_by(username=username.data.strip()).first()
        if existing:
            raise ValidationError("That username is already taken.")

    def validate_email(self, email):
        existing = User.query.filter_by(email=email.data.strip().lower()).first()
        if existing:
            raise ValidationError("That email is already registered.")


class LoginForm(FlaskForm):
    username = StringField("Username", validators=[DataRequired()])
    password = PasswordField("Password", validators=[DataRequired()])
    remember_me = BooleanField("Remember me")
    submit = SubmitField("Log In")
Why This Design Helps
The route can now ask one clean question: form.validate_on_submit(). If validation fails, the form object already knows which fields failed and why.
What does validate_on_submit() actually check?

validate_on_submit() is a convenience method from Flask-WTF. It returns True only when the request is a submitted form request, usually POST, and every validator passes. On a normal GET request, it returns False, which lets the route render the empty form.

05

Build the Registration Route

Registration is the moment where a raw password enters the application. The route should never store that raw password. It should create a User, pass the raw password into the write-only password property, let the model generate a hash, and commit the new user.

Because the route now handles both JSON and form clients, the POST handler branches on request.is_json. The JSON branch is the original API code, unchanged. The form branch relies on form.validate_on_submit(). Adding GET to the allowed methods is what enables rendering the blank form for browser users.

app/auth/views.py — registration route (dual-mode)
python
from flask import request, jsonify, render_template, redirect, url_for, flash
from flask_login import login_user, logout_user, login_required

from app.extensions import db
from app.models import User
from .forms import RegistrationForm, LoginForm
from . import auth_bp


@auth_bp.route("/register", methods=["GET", "POST"])
def register():
    # ── JSON / API path ──────────────────────────────────────────────────
    if request.is_json:
        data     = request.get_json() or {}
        username = data.get("username", "").strip()
        email    = data.get("email",    "").strip().lower()
        password = data.get("password", "")

        if not username or not email or not password:
            return jsonify({"error": "username, email and password are required"}), 400
        if len(password) < 8:
            return jsonify({"error": "password must be at least 8 characters"}), 400
        if User.query.filter_by(username=username).first():
            return jsonify({"error": "username already taken"}), 409
        if User.query.filter_by(email=email).first():
            return jsonify({"error": "email already registered"}), 409

        user = User(username=username, email=email)
        user.password = password
        db.session.add(user)
        db.session.commit()
        return jsonify(user.to_dict()), 201

    # ── HTML form path ───────────────────────────────────────────────────
    form = RegistrationForm()
    if form.validate_on_submit():
        user = User(
            username=form.username.data.strip(),
            email=form.email.data.strip().lower()
        )
        user.password = form.password.data
        db.session.add(user)
        db.session.commit()
        flash("Account created! Please log in.", "success")
        return redirect(url_for("auth.login"))

    return render_template("auth/register.html", form=form)
Talking Points
The route now handles three requests: GET renders the blank form; JSON POST runs the original API logic and returns JSON; form POST runs WTForms validation and redirects. The database write is identical in both POST branches.
06

Build the Login and Logout Routes

Login does not create a new user. It finds an existing user, verifies the submitted password against the stored hash, and then calls login_user(user). Flask-Login stores the user's ID in the signed session cookie, and future requests can access the logged-in user through current_user.

Like registration, the login route branches on request.is_json. The JSON branch returns jsonify(user.to_dict()); the form branch flashes a message and redirects. Logout remains a JSON-only POST endpoint — no browser-rendered page is needed for logging out.

app/auth/views.py — login route (dual-mode) + logout
python
@auth_bp.route("/login", methods=["GET", "POST"])
def login():
    # ── JSON / API path ──────────────────────────────────────────────────
    if request.is_json:
        data     = request.get_json() or {}
        username = data.get("username", "")
        password = data.get("password", "")

        user = User.query.filter_by(username=username).first()
        if user is None or not user.verify_password(password):
            return jsonify({"error": "invalid username or password"}), 401

        remember = data.get("remember_me", False)
        login_user(user, remember=remember)
        return jsonify(user.to_dict()), 200

    # ── HTML form path ───────────────────────────────────────────────────
    form = LoginForm()
    if form.validate_on_submit():
        user = User.query.filter_by(username=form.username.data).first()
        if user is None or not user.verify_password(form.password.data):
            flash("Invalid username or password.", "danger")
            return render_template("auth/login.html", form=form), 401

        login_user(user, remember=form.remember_me.data)
        flash(f"Welcome back, {user.username}! You are now logged in.", "success")
        return redirect(url_for("main_bp.get_recipes"))

    return render_template("auth/login.html", form=form)


@auth_bp.route("/logout", methods=["POST"])
@login_required
def logout():
    logout_user()
    return jsonify({"message": "logged out"}), 200
Security Note
The login failure message is intentionally generic. We should not say "username does not exist" or "password is wrong," because that helps an attacker discover valid usernames.
About Logout
Logout only handles POST and keeps its JSON response. A browser logout button should use a small <form method="POST"> so no JavaScript is needed. The browser receives the JSON response but ignores it — pair this with a next redirect parameter if you want the user to land elsewhere.

Logout changes server-side session state, so it should not be a plain GET link. A small POST form is safer and matches the idea that GET requests should be read-only.
09

Protect Routes That Require a Logged-In User

Public routes like GET /recipes and GET /recipes/<id> remain open to everyone. Routes that write data — POST /recipes, PATCH /recipes/<id>, and DELETE /recipes/<id> — are decorated with @login_required. The PATCH and DELETE routes also check recipe.user_id != current_user.id to enforce ownership, returning a 403 Forbidden if the logged-in user does not own the recipe.

app/routes.py
python
from flask import Blueprint, jsonify, request
from flask_login import login_required, current_user

from .extensions import db
from .models import Recipe

main_bp = Blueprint("main_bp", __name__)


@main_bp.route("/")
def api_home():
    return {"message": "RecipeShare API is running"}


@main_bp.route("/recipes", methods=["GET"])
def get_recipes():
    recipes = Recipe.query.order_by(Recipe.created_at.desc()).all()
    return jsonify([recipe.to_dict() for recipe in recipes])


@main_bp.route("/recipes/<int:recipe_id>", methods=["GET"])
def get_recipe(recipe_id: int):
    recipe = Recipe.query.get_or_404(recipe_id)
    return jsonify(recipe.to_dict())


@main_bp.route("/recipes", methods=["POST"])
@login_required
def create_recipe():
    data = request.get_json() or {}

    required_fields = ["title", "description", "instructions", "prep_time"]
    missing = [field for field in required_fields if field not in data]
    if missing:
        return {"error": f"Missing required fields: {', '.join(missing)}"}, 400

    recipe = Recipe(
        title=data["title"],
        description=data["description"],
        instructions=data["instructions"],
        prep_time=data["prep_time"],
        author=current_user,
    )

    db.session.add(recipe)
    db.session.commit()
    return jsonify(recipe.to_dict()), 201


@main_bp.route("/recipes/<int:recipe_id>", methods=["PATCH"])
@login_required
def update_recipe(recipe_id: int):
    recipe = Recipe.query.get_or_404(recipe_id)
    if recipe.user_id != current_user.id:
        return jsonify({"error": "forbidden"}), 403

    data = request.get_json() or {}
    updatable_fields = ["title", "description", "instructions", "prep_time"]
    for field in updatable_fields:
        if field in data:
            setattr(recipe, field, data[field])

    db.session.commit()
    return jsonify(recipe.to_dict()), 200


@main_bp.route("/recipes/<int:recipe_id>", methods=["DELETE"])
@login_required
def delete_recipe(recipe_id: int):
    recipe = Recipe.query.get_or_404(recipe_id)
    if recipe.user_id != current_user.id:
        return jsonify({"error": "forbidden"}), 403
    db.session.delete(recipe)
    db.session.commit()
    return "", 204
Authentication + Authorization
@login_required checks that someone is logged in. The recipe.author_id != current_user.id check handles authorization: this logged-in user may not be allowed to modify every record.

Test the Form-Based Flow

Form-based routes can be tested with Flask’s test client. In tests, CSRF is often disabled so we can focus on route behavior, validation rules, password hashing, and session changes. The tests should verify successful registration, duplicate username validation, successful login, bad login, and protected route behavior.

tests/test_auth_forms.py
python
import pytest

from app import create_app, db
from app.models import User


@pytest.fixture
def client():
    app = create_app({
        "TESTING": True,
        "WTF_CSRF_ENABLED": False,
        "SQLALCHEMY_DATABASE_URI": "sqlite:///:memory:",
        "SECRET_KEY": "test-secret",
    })

    with app.app_context():
        db.create_all()
        yield app.test_client()
        db.drop_all()


def register(client, username="alice", email="alice@example.com", password="password123"):
    return client.post("/auth/register", data={
        "username": username,
        "email": email,
        "password": password,
        "confirm_password": password,
    }, follow_redirects=True)


def login(client, username="alice", password="password123"):
    return client.post("/auth/login", data={
        "username": username,
        "password": password,
    }, follow_redirects=True)


def test_register_creates_user_with_hashed_password(client):
    response = register(client)
    assert response.status_code == 200

    user = User.query.filter_by(username="alice").first()
    assert user is not None
    assert user.password_hash != "password123"
    assert user.verify_password("password123") is True


def test_register_rejects_duplicate_username(client):
    register(client)
    response = register(client, email="other@example.com")
    assert b"already taken" in response.data


def test_login_success(client):
    register(client)
    response = login(client)
    assert b"logged in" in response.data.lower()


def test_login_wrong_password(client):
    register(client)
    response = login(client, password="wrong-password")
    assert response.status_code == 401
    assert b"invalid username or password" in response.data.lower()
Testing Goal
We are not testing Flask-WTF itself. We are testing that our application wires the form, model, database session, and Flask-Login behavior together correctly.

End-of-Lesson Checklist

By the end of this lesson, we should be able to talk through the complete browser-based authentication flow.

  • We can explain the difference between API authentication and form-based authentication.
  • We can create Flask-WTF form classes with validators for username, email, password, and password confirmation.
  • We can render forms in templates, including {{ form.hidden_tag() }} for CSRF protection.
  • We can register a user without storing the plain-text password.
  • We can verify a submitted password using the stored password hash.
  • We can call login_user() and logout_user() to manage user sessions.
  • We can use current_user to customize the navbar and connect created records to the logged-in user.
  • We can protect routes with @login_required and separately check record ownership.
  • We can use request.is_json to serve both API and form clients from a single route without breaking existing JSON behavior.
  • We can test registration and login routes using Flask's test client.
Big Picture
The core authentication system is now usable from a browser and the existing JSON API continues to work unchanged. The same route serves both clients by branching on request.is_json. This pattern — extend, don't replace — is common in apps that are transitioning from a pure API toward a full-stack architecture.
