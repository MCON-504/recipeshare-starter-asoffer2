What is User Authentication?

Most web applications need to know who is making a request before granting access to protected resources. User authentication is the process of verifying a user's identity — confirming that someone is who they claim to be.

In our Recipe Share app, authentication lets us associate recipes with their authors, prevent anonymous edits, and eventually support features like saved recipes or private notes.

High-level login flow

📌 Key Concept
Authentication answers "Who are you?" — it's separate from authorization, which answers "What are you allowed to do?" We'll tackle authorization (roles & permissions) in a later lesson.

Password Hashing & Salting

Never store plain-text passwords. If your database is ever compromised, an attacker would have every user's password instantly. Instead, we store a hash — a one-way mathematical transformation of the password.

Password + salt → one-way hash function → stored hash. Salting ensures two users with the same password get different hashes.

Why Salt?

A salt is a randomly generated string added to the password before hashing. Without salts, attackers can use rainbow tables — precomputed hash dictionaries — to reverse common passwords instantly. Salts make every hash unique, even if two users share the same password.

✅ Good News
Werkzeug (Flask's underlying library) handles salting automatically via generate_password_hash(). You never need to manage salts manually.
🧂 Deep Dive — What exactly is a salt?

A salt is a random string generated fresh for each user at the moment their password is hashed. It is combined with the password before the hash function runs, so two users with the same password produce completely different hashes.

conceptual example
text
User A  →  "hello123" + salt "7f3a9b2c"  →  hash: aabbcc...
User B  →  "hello123" + salt "d92f1e4a"  →  hash: xxyyzz...
                                                     ↑ completely different!

Why not just hash without a salt?

Without a salt, every user with the password "hello123" produces the same hash. An attacker who steals the database can use a rainbow table — a precomputed lookup of hash → password for millions of common passwords — and crack every matching account instantly.

With a salt, the rainbow table is useless. The attacker would need to build a separate table for every possible salt value — computationally infeasible.

Where is the salt stored?

The salt lives inside the hash string itself — no separate database column needed. Werkzeug encodes everything together:

anatomy of a Werkzeug hash string
text
pbkdf2:sha256$260000$7f3a9b2c$a1b2c3d4e5f6...
              ↑        ↑         ↑          ↑
           algorithm  iterations  salt    actual hash

When a user logs in, check_password_hash() reads the salt back out of that string, re-hashes the submitted password with the same salt, and compares — all automatically. You never touch the salt directly.

⚠️ Salt ≠ Pepper
A pepper is a secret value (stored outside the database, e.g. in an environment variable) added on top of the salt for an extra layer of security. Werkzeug does not add a pepper automatically — that's an advanced hardening step beyond this lesson.

Werkzeug Password Helpers

python — interactive example
python
from werkzeug.security import generate_password_hash, check_password_hash

# Registration — hash before storing
password_hash = generate_password_hash("p@ssw0rd!")
print(password_hash)
# → 'pbkdf2:sha256:260000$7f3a9b2c...$a1b2c3...'

# Login — verify without reversing the hash
check_password_hash(password_hash, "p@ssw0rd!")   # → True
check_password_hash(password_hash, "wrongpass")   # → False
🔬 Deep Dive — PBKDF2 vs bcrypt (what are we actually using?)

Definition: PBKDF2: Password-Based Key Derivation Function 2

Werkzeug's generate_password_hash() defaults to PBKDF2:sha256. You may hear about bcrypt as an alternative. Here's how they compare:

PBKDF2 (what we use) bcrypt
Algorithm Applies SHA-256 thousands of times ("key stretching") Uses the Blowfish cipher with a tunable cost factor
Work factor 260,000 iterations (Werkzeug default) Cost factor 10–12 (each +1 doubles the work)
GPU resistance MODERATE — can be parallelised on GPUs BETTER — memory-hard, harder to parallelise
Standard NIST / FIPS approved ✅ Not FIPS approved
Password length limit None 72 bytes max
Dependencies None — built into Python's stdlib Requires bcrypt package
In this project? ✅ Yes — via werkzeug.security ❌ No
📌 Bottom Line
For a student project, PBKDF2:sha256 is perfectly fine — it's battle-tested and requires zero extra packages. In a production app handling sensitive data you'd evaluate bcrypt or the even stronger Argon2 (winner of the 2015 Password Hashing Competition). The key insight is the same for all three: they are deliberately slow so that brute-force attacks take impractically long.

Flask-Login & UserMixin

Flask-Login manages user sessions after authentication. It tracks who is logged in, handles the current_user proxy, and provides the @login_required decorator. To use it, our User model must implement three properties and one method:

Inheriting UserMixin provides the four required Flask-Login properties with sensible defaults.

Setting Up Flask-Login in the App Factory

The project separates extension instances into extensions.py so they can be imported anywhere without circular imports, then initialised inside the app factory.

app/extensions.py
python
from flask_sqlalchemy import SQLAlchemy
from flask_migrate import Migrate
from flask_login import LoginManager

# Extensions are created here and initialized in the app factory.
db            = SQLAlchemy()
migrate       = Migrate()
login_manager = LoginManager()
app/__init__.py
python
from flask import Flask

from .config import Config
from .extensions import db, migrate, login_manager


def create_app(test_config: dict | None = None) -> Flask:
    app = Flask(__name__)
    app.config.from_object(Config)

    # Allow tests to override config before extensions bind
    if test_config:
        app.config.update(test_config)

    db.init_app(app)
    migrate.init_app(app, db)

    # Flask-Login setup
    login_manager.login_view = "auth.login"
    login_manager.login_message_category = "warning"
    login_manager.init_app(app)

    from .routes import main_bp
    app.register_blueprint(main_bp, url_prefix="/api")

    from .auth import auth_bp
    app.register_blueprint(auth_bp, url_prefix="/auth")

    return app


# Flask-Login needs this to reload a user from the session
@login_manager.user_loader
def load_user(user_id: str):
    from .models import User
    return User.query.get(int(user_id))
🔑 Secret Key
The SECRET_KEY signs the session cookie. In production, load it from an environment variable — never hard-code it in source. Use os.environ.get("SECRET_KEY").

Updating the User Model

Our existing User model stores username and email. We need to add a password_hash column and expose helper methods so the rest of the app never interacts with raw passwords.

app/models/user.py
python
from app import db
from werkzeug.security import generate_password_hash, check_password_hash
from flask_login import UserMixin


class User(UserMixin, db.Model):
    __tablename__ = "users"

    id            = db.Column(db.Integer, primary_key=True)
    username      = db.Column(db.String(80),  nullable=False, unique=True)
    email         = db.Column(db.String(120), nullable=False, unique=True)
    password_hash = db.Column(db.String(256), nullable=False)

    recipes = db.relationship("Recipe", back_populates="author", lazy=True)

    # ── password property (write-only) ──────────────────
    @property
    def password(self):
        raise AttributeError("password is write-only")

    @password.setter
    def password(self, raw_password: str):
        self.password_hash = generate_password_hash(raw_password)

    # ── verify_password ──────────────────────────────────
    def verify_password(self, raw_password: str) -> bool:
        return check_password_hash(self.password_hash, raw_password)

    # ── to_dict (excludes hash for safety) ──────────────
    def to_dict(self) -> dict:
        return {
            "id":       self.id,
            "username": self.username,
            "email":    self.email,
        }
⚠️ Write-Only Pattern
Raising AttributeError on the password getter is intentional. It prevents accidental exposure of the hash and makes it clear that user.password is an input slot, not something you can read back.

Registration Route

The registration endpoint accepts a JSON body, validates that the username and email are not already taken, hashes the password, and persists the new user.

app/auth/views.py — register
python
from flask import request, jsonify
from app import db
from app.auth import auth_bp
from app.models.user import User


@auth_bp.route("/register", methods=["POST"])
def register():
    data = request.get_json()
    username = data.get("username", "").strip()
    email    = data.get("email", "").strip().lower()
    password = data.get("password", "")

    # Basic validation
    if not username or not email or not password:
        return jsonify({"error": "username, email and password are required"}), 400

    if len(password) < 8:
        return jsonify({"error": "password must be at least 8 characters"}), 400

    # Check for duplicates
    if User.query.filter_by(username=username).first():
        return jsonify({"error": "username already taken"}), 409

    if User.query.filter_by(email=email).first():
        return jsonify({"error": "email already registered"}), 409

    # Create user — password property calls generate_password_hash internally
    user = User(username=username, email=email)
    user.password = password          # uses the setter we defined
    db.session.add(user)
    db.session.commit()

    return jsonify(user.to_dict()), 201

Login & Logout Routes

Login flow: credentials → password verification → session cookie → response

app/auth/views.py — login & logout
python
from flask_login import login_user, logout_user, login_required


@auth_bp.route("/login", methods=["POST"])
def login():
    data     = request.get_json()
    username = data.get("username", "")
    password = data.get("password", "")

    user = User.query.filter_by(username=username).first()

    if user is None or not user.verify_password(password):
        # Same error for both cases — don't hint which field was wrong
        return jsonify({"error": "invalid username or password"}), 401

    remember = data.get("remember_me", False)
    login_user(user, remember=remember)
    return jsonify(user.to_dict()), 200


@auth_bp.route("/logout", methods=["POST"])
@login_required
def logout():
    logout_user()
    return jsonify({"message": "logged out"}), 200
🔒 Security Note — Enumeration
Always return the same error message whether the username doesn't exist or the password is wrong. Separate messages let attackers enumerate valid usernames (user enumeration attack).

Protected Routes

Any route that should only be accessible to logged-in users gets the @login_required decorator. If an unauthenticated request arrives, Flask-Login redirects to the login_view we configured ("auth.login").

app/main/views.py — protected recipe endpoints
python
from flask import request, jsonify
from flask_login import login_required, current_user
from app import db
from app.main import main_bp
from app.models.recipe import Recipe


# ── Public: anyone can browse recipes ──────────────────
@main_bp.route("/recipes", methods=["GET"])
def list_recipes():
    recipes = Recipe.query.all()
    return jsonify([r.to_dict() for r in recipes])


# ── Protected: only authenticated users can create ─────
@main_bp.route("/recipes", methods=["POST"])
@login_required
def create_recipe():
    data = request.get_json()
    recipe = Recipe(
        title=data["title"],
        body=data["body"],
        author=current_user,   # Flask-Login provides this proxy
    )
    db.session.add(recipe)
    db.session.commit()
    return jsonify(recipe.to_dict()), 201


# ── Protected: only the author can delete their recipe ──
@main_bp.route("/recipes/<int:recipe_id>", methods=["DELETE"])
@login_required
def delete_recipe(recipe_id):
    recipe = Recipe.query.get_or_404(recipe_id)
    if recipe.author_id != current_user.id:
        return jsonify({"error": "forbidden"}), 403
    db.session.delete(recipe)
    db.session.commit()
    return "", 204

Testing the User Model

import pytest
from app.models.user import User


def test_password_setter_hashes():
    """Setting password must store a hash, not plain text."""
    user = User(username="alice", email="alice@example.com")
    user.password = "s3cret!"
    assert user.password_hash is not None
    assert user.password_hash != "s3cret!"


def test_password_getter_raises():
    """Reading .password must raise AttributeError."""
    user = User(username="bob", email="bob@example.com")
    user.password = "s3cret!"
    with pytest.raises(AttributeError):
        _ = user.password


def test_verify_password_correct():
    """verify_password returns True for the correct password."""
    user = User(username="carol", email="carol@example.com")
    user.password = "mypassword"
    assert user.verify_password("mypassword") is True


def test_verify_password_wrong():
    """verify_password returns False for wrong password."""
    user = User(username="dave", email="dave@example.com")
    user.password = "mypassword"
    assert user.verify_password("wrongpassword") is False


def test_two_users_same_password_different_hashes():
    """Salting must produce different hashes for the same password."""
    u1 = User(username="eve",   email="eve@example.com")
    u2 = User(username="frank", email="frank@example.com")
    u1.password = u2.password = "samepass"
    assert u1.password_hash != u2.password_hash

Best Practices Checklist

  • Store passwords as hashes — never plain text.
  • Use generate_password_hash / check_password_hash from Werkzeug — don't roll your own.
  • Enforce a minimum password length (≥ 8 characters) on the server side.
  • Return identical errors for bad username vs bad password to prevent enumeration.
  • Load SECRET_KEY from an environment variable in production.
  • Use HTTPS in production to protect the session cookie in transit.
  • Consider adding account lockout after repeated failed logins to thwart brute force.
